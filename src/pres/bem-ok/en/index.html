BEM is okay

First time I heard about BEM 7 years ago from Yandex developers. I was never against it, but I thought about it as a big overhead. And only last year I actually tried it realized that it’s not just “okay”, it’s really good.

My name is Vadim Makeev, I work for Opera Software in Developer Relations team in Oslo. But this talk is not about our great products, it’s about technology as I’m really passioned for front-end development and I want to share my passion and experience.

Web development is not just about creating something new from scratch. Bigger part of it is support and modification of existing code, not just “your code”, but also “their code”. That’s why it’s important to have methodology and code style.

This is your web site right after launch: it’s healthy, shiny and beautiful. In most cases this is how it will look after a year of “content management” and further development. After another year it will probably look like this: just a bloody mess instead of a former beauty.

Is CSS that bad?

			H1 { color: blue }
			P EM { font-weight: bold }
			A:link IMG { border: 2px solid blue }
			A:visited IMG { border: 2px solid red }
			A:active IMG { border: 2px solid lime }

From CSS 1 specification

Why is it so? Not because CSS is bad, it’s because sometimes we use it as in early 90s. This is not just a bad CSS from someone’s code, this is example from CSS 1 specification from 1994. CSS is used to be like this, but it could be much better.

So what?

I read two great articles last year. They really changed the way I work with CSS. I recommend you to read them, but in this specific order.

Cargo Cult CSS

First: Cargo Cult CSS, claiming that BEM is the worst thing that happened with CSS and CSS is mostly “okay” with all :nth-something > :last-child::before things.

CSS Architecture

Second: CSS Architecture, extending programming principles to CSS using BEM methodology. The best BEM introduction I’ve ever read.

History

BEM is not something completely new. There were other people trying to make CSS more stable, bulletproof and maintainable. Let’s briefly have a look at those attempts.

As we’re talking about history and web standards we should mention this guy in a blue beanie. Thank you Jeffrey, for everything.

Dan
Cederholm

Dan Cederholm wrote the great “Bulletproof Web Desing” book back 2005. Some ideas of this book are still relevant today.

I beleive it was the first well known attempt to start thinking about front-end code quality.

Nicole
Sullivan

Then in 2009 Nicole Sullivan came up with OOCSS idea.

OOCSS

OOCSS sounds bit naive from today’s point of view but it was important step in the right direction.

Jonathan
Snook

Two years ago 2012 Jonathan Snook published his SMACSS book based on idea of modular and reusable CSS.

SMACSS

You probably saw this nice fellow with an axe. I would call this idea as “BEM for beginners”. It’s not that strict and could fit your needs better.

As you’re probably already heard Web Components is the next big thing. It’s all about components incapsulation, pretty similar to what BEM is already doing for a last 5 years alredy.

Vitaly
Harisov

Seven years ago in 2007 Vitaly Harisov (head of front-end development in Yandex for may years) came up with idea of absolutely independent blocks.

BEM

Two years later BEM idea got its name and basic conventions. It was 5 years ago.

Block
Element
Modifier

BEM stands for: Block, Element, Modifier.

1. No IDs
2. No cascade
3. Unique classes

And consists of a number of rules, here are the most important: no IDs, no cascase (for Cascading Style Sheets, yeah) and only unique classes are allowed.

BEM

			.b-block {
				}
				.b-block__element {
					}
				.b-block_key_value {
					}

Basically it look like this: block and its elements and modifiers, all prefixed with b- and block name, which makes every class name unique.

bem.info

From Yandex’s point of view BEM is not just a CSS convention, it’s also template engine, JavaScript library, tools, server and many other things. You can learn more on bem.info. But today I’m talking mostly about BEM as the basic idea of naming UI elements and its states.

Harry
Roberts

Nicolas
Gallagher

Two years this two guys fell in love with BEM idea and interpreted it in a new way that became quite popular.

BEM

It’s not completely new BEM, it’s rather a few adjustments that get this idea closer to people.

4. No b-
5. __ & --

In addition to the original principles of BEM this interpretation adds two new: there’s no need in b- plus new, more consistent and easier to read element and modifier separators.

BEM

			.block {
				}
				.block__element {
					}
				.block--modifier {
					}

It’s pretty much the same BEM but with two underscores for element and two dashes for modifier.

csswizardry-grids

The great example library based on BEM idea is csswizardry-grids, modular grids library for Sass written by Harry Roberts. Everything you do with this library is basically eiter grid element or its elements and modifiers.

Stylitis

There’s a well known disease called stylitis, it was a big problem a while ago.

Stylitis

			<span style="display: block; font-family: Arial, sans-serif;
				font-size: 11px; color: blue; border-style: solid;
				border-color: blue; border-width: 2px; padding: 20px;">
				Terrible!
			</span>

The main symptom if this disease is style attribute full of inline style. Terrible, right? We all agree.

Classitis

But there’s another similar problem nowadays: classitis.

Classitis

			<span class="display-block blue-box
				font-arial color-blue
				solid-blue-border padding-20">
				Beautiful?
			</span>

Symptoms look pretty similar to stylitis: a lot classed serving presentation is the same terrible way. It’s really hard to maintain. And you also start suffering from this disease using BEM.

Header CSS

			.header {
				}
				.header__logo {
					}
					.header__logo--huge { … }
					.header__logo--monochome { … }
					.header__logo--upside-down { … }

Let’s have a look at typical BEM element: header with search element which have a number of modificators. “Huge” for bigger logo, “monochrome” for print version and “upside-down” for a April 1st jokes.

Header HTML

			<div class="header>
				<div class="
					header__logo
					header__logo--huge
					header__logo--monochome
					header__logo--upside-down">

But if one day you will have to build print version of a huge logo for April 1st joke you’ll end up with following markup. This is definitely a stylitis, thust me I’m a doctor.

Weird

This is obvious downside of using BEM idea.

Preprocessors

But it could be easily solved by preprocessors.

CSS

			.block {
				}
				.block__element {
					}
				.block--modifier {
					}

Here’s our typical BEM styles: block, elemen, modifier. As you can see we have to repeat block name over and over again.

Sass

			.block {
				&__element {
					}
				&--modifier {
					}
				}

But with Sass magic you can use ampersand to prepend block’s name to every nested element of modifier. But they will not be actually nested in output CSS as typical nested selector.

Sass

			.block {
				&__element {
					&--modifier { … }
					}
				&--modifier {
					}
				}

You can even nest modifiers to elements and resulted class will be also prepended with block and element names. Now you can easily change block name.

Classitis cured

			.header {
				&__logo {
					&--perverted {
						@extend %huge;
						@extend %monochrome;
						@extend %upside-down;

Let’s get back to our header block with search. This how we can cure symptoms of classitis: by adding new abstraction level using Sass silent classes.

Classitis cured

			<div class="header>
				<div class="
					header__logo
					header__logo--huge
					header__logo--monochome
					header__logo--upside-down">

Before.

Common sense

			<div class="header>
				<div class="
					header__logo
					header__logo--perverted">

After. Much better, right?

What else?

BEML

There’s a similar solution for HTML called BEML that allows you not to write huge classes in HTML.

BEML

			<header block="header">
				<input elem="search" mod="size: huge">
			</header>

You just use bloc, elem and mod attributes that will be converted to actual BEM classes.

Modifiers

			styles/
				catalog.css
				catalog__layout.css
		
			pictures/
				picture.jpg
				picture--thumb.jpg

You can also use this double underscored and double dashes approach for naming anything you want: not just component files, but also picture thumbnails, for example.

Sublime Text

			{
				"word_separators":
				"./\\()\"'-_:,.;<>~!@#$%^&*|+=[]{}`~?"
			}

There’s a little trick to work with underscores easier: add underscore symbol as a word separator in your editor options. It will help you to navigate in such selectors much faster.
